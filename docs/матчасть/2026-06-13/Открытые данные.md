# Открытые данные и платформа v0.1 «Цифрового двойника России»

Документ отвечает на три прикладных вопроса, поставленных перед этапом акселерации:

1. Есть ли уже сейчас открытые данные, которых достаточно для открытого контура «Цифрового двойника России»?
2. Что из этих данных можно собрать в платформу версии 0.1 для презентации возможностей двойника?
3. Какие первые шаги нужны для запуска двойника на этапе акселерации проекта.

Документ опирается на материалы [Описание](Описание.md), [MVP](MVP.md), [Заявка на акселерацию](Заявка%20на%20акселерацию.md), [Чаво](Чаво.md) и [Примеры обработки информации для контуров](Примеры%20обработки%20информации%20для%20контуров.md). Все источники указаны по состоянию на май 2026 года и подлежат повторной проверке перед использованием — состав открытых наборов в РФ заметно менялся в 2023–2025 годах.

---

## 1. Краткий ответ

**Да, открытых данных для запуска открытого контура достаточно.** Их хватает на то, чтобы за 2–3 месяца силами небольшой команды собрать платформу версии 0.1 на двух пилотных регионах и публично показать ключевые сценарии двойника — региональный паспорт, дефициты кадров, логистические уязвимости и базовый риск ЧС.

При этом важно сразу зафиксировать два ограничения:

- Платформа v0.1 опирается только на **открытые** и **синтетические** данные. Подключение ведомственных слоёв возможно только после появления государственного оператора пилота и заключения соглашений (см. вопросы 2 и 5 [FAQ](Чаво.md)).
- Часть открытых наборов в 2023–2025 годах была сокращена или перестала обновляться. Поэтому состав данных в v0.1 фиксируется на конкретный месяц и сопровождается паспортом качества для каждого набора.

## 2. Какие открытые данные доступны прямо сейчас

Ниже сведены источники, реально доступные на май 2026 года и пригодные для открытого контура двойника. Для каждого источника указан тип данных и форма доступа.

### 2.1. Геопространственная основа

| Источник | Что даёт | Форма доступа |
|----------|----------|---------------|
| OpenStreetMap (Россия), выгрузки Geofabrik | Дорожная сеть, здания, точки интереса, гидрография, ж/д, административные границы | Свободные дампы `.osm.pbf`, лицензия ODbL |
| NextGIS Data | Региональные выгрузки на основе OSM | Свободные и платные слои, GeoJSON/Shapefile |
| Natural Earth | Низкодетальные слои для контекста (границы, реки, побережья) | Public domain |
| НСПД (nspd.gov.ru) | Публичная кадастровая карта, агрегированные пространственные слои | Веб-просмотр и тайловые сервисы, ограниченная массовая выгрузка |
| ГАР (бывший ФИАС, fias.nalog.ru) | Государственный адресный реестр: улицы, дома, индексы | Полные дампы XML/DBF |

Этого блока достаточно, чтобы у двойника появилась полноценная базовая карта по любому региону РФ.

### 2.2. Экономика и предприятия

| Источник | Что даёт | Форма доступа |
|----------|----------|---------------|
| Росстат (rosstat.gov.ru/opendata) | Промышленность, торговля, услуги, цены, региональные индикаторы | CSV/XLSX, открытая лицензия |
| ЕМИСС (fedstat.ru) | Унифицированные межведомственные показатели | Веб-интерфейс, открытые наборы, неофициальные API |
| ФНС: открытые данные (nalog.gov.ru/opendata) | Численность работников (>100), задолженности, спецрежимы | CSV/XML |
| ГИР БО (bo.nalog.gov.ru) | Бухгалтерская отчётность организаций | Бесплатный поиск по организации, ограниченный массовый доступ |
| ЕГРЮЛ/ЕГРИП (egrul.nalog.ru) | Реквизиты юридических лиц и ИП | Поиск по запросу, массовая выгрузка — по подписке |
| zakupki.gov.ru, spending.gov.ru, bus.gov.ru | Государственные закупки, расходы, госучреждения | Открытые дампы, FTP |

Эти наборы позволяют построить карту экономической активности региона: где сосредоточены предприятия, какие отрасли преобладают, какие закупки и расходы идут на территории.

### 2.3. Рынок труда и образование

| Источник | Что даёт | Форма доступа |
|----------|----------|---------------|
| «Работа России» (trudvsem.ru/opendata/api) | Открытое API вакансий и резюме (где допустимо законом) | REST/JSON, лимиты на запрос |
| Росстат | Демография, занятость, миграция | CSV, открытые таблицы |
| Минпросвещения (opendata.edu.gov.ru) | Реестр школ и программ | CSV/XML |
| Минобрнауки (minobrnauki.gov.ru/opendata) | Вузы, аккредитованные программы, контрольные цифры приёма | CSV |

Эта подборка закрывает один из ключевых сценариев MVP — поиск кадровых дефицитов и сопоставление вакансий с реальными образовательными программами.

### 2.4. Природные ресурсы, экология и ЧС

| Источник | Что даёт | Форма доступа |
|----------|----------|---------------|
| Росгидромет (meteorf.gov.ru/opendata, meteo.ru) | Климатические и гидрологические наблюдения, опасные явления | CSV/XML, веб-сервисы |
| МЧС (mchs.gov.ru/opendata) | Реестры ЧС, объекты помощи в разрешённом виде | CSV |
| NASA FIRMS | Действующие очаги пожаров (MODIS/VIIRS) по территории РФ | REST API, CSV/SHP/WMS, бесплатный ключ |

| Рослесхоз, Росводресурсы (открытые публикации) | Лесной реестр, водохозяйственные данные в открытой части | Веб-публикации, ограниченные дампы |

В сумме это даёт открытый «природно-экологический» слой: пожары, гидрология, климатические наблюдения и контекст лесов и водоёмов.

### 2.5. Транспорт и инфраструктура (открытая часть)

| Источник | Что даёт | Форма доступа |
|----------|----------|---------------|
| OSM (дороги, ж/д, мосты, объекты транспорта) | Базовая транспортная сеть | Дампы Geofabrik |
| Минтранс, Росавтодор, региональные органы (открытые публикации) | Проекты развития сети, ремонты | Веб-публикации |
| СО ЕЭС (so-ups.ru, открытые сводки) | Региональные показатели нагрузки и генерации в агрегированной форме | Веб-публикации |
| Минэнерго (minenergo.gov.ru/opendata) | Открытые реестры и отчёты | CSV/XML |
| zakupki.gov.ru | Инфраструктурные закупки и контракты | Открытые дампы |

Этого достаточно, чтобы показать транспортный каркас региона и базовые ограничения, но без детальных схем сетей. Чувствительные сетевые данные в v0.1 не подключаются ни в каком виде — они относятся к инфраструктурному и закрытому контурам (см. вопрос 6 [FAQ](Чаво.md)).

### 2.6. Региональные и городские порталы

| Источник | Что даёт | Форма доступа |
|----------|----------|---------------|
| data.mos.ru | >1000 наборов по Москве: транспорт, экология, инфраструктура, здравоохранение, образование | API по токену, CSV/JSON |
| data.gov.spb.ru | Открытые данные Санкт-Петербурга | API по токену, CSV/JSON |
| Региональные порталы открытых данных (Татарстан, Тюмень, Свердловская обл. и др.) | Локальная инфраструктура, экология, услуги | CSV/XML, индивидуальные форматы |

Один из этих регионов целесообразно сделать первой пилотной территорией для v0.1: уровень готовности данных там выше, чем в среднем по стране.

### 2.7. Что **не** входит в v0.1 принципиально

В открытом контуре платформы версии 0.1 **не используются**:

- персональные данные граждан в любом виде;
- детальные схемы и режимы критической инфраструктуры;
- идентификаторы микропредприятий, по которым возможна реидентификация;
- любые данные ограниченного доступа без согласования с уполномоченным органом;
- закрытые сценарии устойчивости, мобилизационные и оборонные сведения.

Эти ограничения следуют из требований к открытому контуру, описанных в [MVP](MVP.md) и в разделах 1, 3, 6 [FAQ](Чаво.md).

### 2.8. Ограничения открытых данных, которые надо учитывать

- **Фрагментарность.** Единого федерального каталога открытых данных в полноценном виде нет: data.gov.ru был перезапущен в 2025 году, но наполнение пока ограничено. Большая часть наборов публикуется напрямую на сайтах ведомств в виде JSON/CSV.
- **Свежесть.** Многие наборы обновляются раз в год или нерегулярно. В каждом паспорте набора данных в v0.1 указывается дата последней загрузки.
- **Сокращение наборов в 2023–2025 годах.** Часть демографических и инфраструктурных показателей перестала обновляться. Сценарии v0.1 рассчитаны с учётом этих ограничений и сопровождаются оговорками о неполноте.
- **Форматы.** Преобладают CSV (часто в кодировке Windows-1251), XLSX, XML. Геоданные нередко поступают в Shapefile или MapInfo, реже в GeoJSON. ETL-пайплайн v0.1 обязан корректно обрабатывать эти форматы.
- **Лимиты массового доступа.** Часть наборов (ЕГРЮЛ, ГИР БО) ограничивает массовую выгрузку. Для v0.1 это не критично: открытый контур работает на агрегированных и обезличенных показателях, а не на массовой выгрузке записей.

## 3. Что можно показать в платформе версии 0.1

Платформа v0.1 — это **демонстрационный прототип открытого контура** двойника на одном-двух пилотных регионах. Её цель — не заменить полноценный MVP, а наглядно показать жизнеспособность концепции на реальных открытых данных и стать опорой для презентаций в рамках акселерации.

### 3.1. Состав платформы v0.1

| Блок | Минимальное наполнение в v0.1 |
|------|--------------------------------|
| Регионы | 1–2 пилотных региона: один с развитой открытой инфраструктурой данных (например, Москва или Санкт-Петербург), один с типовой региональной выкладкой |
| Базовая карта | Векторные тайлы на основе OSM (Geofabrik) с административными границами и базовой инфраструктурой |
| Слои | Транспортная сеть, объекты социальной инфраструктуры, основные предприятия, точки занятости, экологические объекты, очаги пожаров |
| Аналитика | Региональный паспорт, дефицит кадров по открытым вакансиям, сравнение площадок, базовая оценка ЧС-риска |
| Контур | Только открытый, с агрегацией до уровня муниципалитета/района |
| Интерфейс | Веб-карта, фильтры по слоям, карточки объектов, экспорт отчёта в PDF/Markdown |
| API | Чтение разрешённых слоёв и выгрузка отчёта |
| Доступ | Публичный, без обязательной регистрации; журналирование запросов в обезличенном виде |
| Документация | Паспорт каждого набора данных, инструкция пользователя, ограничения и оговорки |

Это согласуется с границами MVP (см. [MVP](MVP.md), раздел «Что входит в MVP») и сознательно сужает его до открытого контура.

### 3.2. Демонстрационные сценарии v0.1

Для презентации достаточно четырёх сценариев, каждый из которых строится только на открытых данных:

1. **Региональный паспорт.** Пользователь выбирает регион и получает карту базовой инфраструктуры, агрегированные показатели по экономике, занятости, демографии и экологии. Источники: OSM, Росстат, ЕМИСС, ФНС открытые наборы, региональные порталы.
2. **Кадровый дефицит.** Сопоставление открытых вакансий «Работы России» с агрегированной демографией и образовательными программами Минпросвещения/Минобрнауки. Результат — карта дефицита по профессиям и муниципалитетам.
3. **Сравнение площадок для проекта.** Пользователь отмечает несколько потенциальных площадок и получает сравнение по транспортной доступности (OSM), наличию рабочей силы (Росстат + «Работа России») и базовой экологической нагрузке (NASA FIRMS, Росгидромет).
4. **Базовый риск ЧС.** Карта действующих и исторических очагов пожаров (NASA FIRMS), опасных гидрометеорологических явлений (Росгидромет), наложенная на транспортную сеть и социальную инфраструктуру.

В каждой панели обязательно показывается источник, дата выгрузки, лицензия и оговорки о неполноте — это основной приём, который отличает v0.1 от рекламной демонстрации.

### 3.3. Архитектура v0.1

```
Открытые источники (OSM, Росстат, ЕМИСС, «Работа России», NASA FIRMS, Copernicus, региональные порталы)
        ↓
ETL-пайплайн: загрузка, нормализация, паспорт качества
        ↓
PostgreSQL/PostGIS + объектное хранилище + каталог метаданных
        ↓
FastAPI: API чтения и сервис расчёта простых сценариев
        ↓
Веб-карта на MapLibre GL JS + аналитические панели + экспорт отчётов
        ↓
Пользователи: участники акселератора, эксперты, потенциальные операторы пилота
```

Технологический контур v0.1 — подмножество стека MVP:

- **PostgreSQL/PostGIS** — пространственная база;
- **Python/FastAPI** — API и расчёт сценариев;
- **MapLibre GL JS** + векторные тайлы из OSM — карта;
- **React или Vue** — фронтенд;
- **dbt/Airflow или скрипты** — расписание загрузок;
- **Keycloak или аналог** — заготовка под ролевую модель (в v0.1 публичный доступ + журналирование).

Стек полностью совместим с целевой архитектурой MVP, что позволяет переиспользовать код без переписывания.

### 3.4. Что v0.1 явно **не** делает

- Не работает с ведомственными, инфраструктурными и закрытыми контурами.
- Не подключает чувствительные данные критической инфраструктуры.
- Не претендует на федеральное покрытие — только пилотные регионы.
- Не подменяет существующие государственные системы (НСПД, ЕМИСС, ГИСП и т. п.).
- Не выдаёт автоматических управленческих решений — только аналитику и сценарии.

Эти ограничения зафиксированы в [MVP](MVP.md) («Что не входит в MVP»), и v0.1 их строго соблюдает.

## 4. Первые шаги по реализации двойника на этапе акселерации

Цель этапа акселерации — пройти путь TRL 2–3 → TRL 4–5: от концепции и обращений к лабораторному, а затем пилотному прототипу на 1–2 регионах ([Заявка на акселерацию](Заявка%20на%20акселерацию.md), раздел 5). Платформа v0.1 — это первый рабочий артефакт этого пути.

### 4.1. Дорожная карта первых шагов (90 дней)

| Шаг | Срок | Результат |
|-----|------|-----------|
| 1. Уточнение пилотных регионов и сценариев | 1-я неделя | Утверждены 1–2 региона и 4 демонстрационных сценария |
| 2. Каталогизация открытых источников | 1–2-я недели | Список наборов с паспортами качества, лицензиями и датами |
| 3. Поднятие инфраструктуры (PostGIS, объектное хранилище, CI) | 1–3-я недели | Работающий стенд для приёма данных |
| 4. ETL для базовых слоёв (OSM, Росстат, ЕМИСС, «Работа России», NASA FIRMS) | 2–6-я недели | Загруженные нормализованные слои с паспортами |
| 5. Веб-карта и карточки объектов | 4–8-я недели | Рабочая карта с фильтрами и поиском |
| 6. Сценарии «Региональный паспорт» и «Кадровый дефицит» | 6–10-я недели | Две панели на реальных данных |
| 7. Сценарии «Сравнение площадок» и «Базовый риск ЧС» | 8–11-я недели | Ещё две панели |
| 8. Экспорт отчётов и API чтения | 9–11-я недели | Отчёт в PDF/Markdown + публичный API |
| 9. Внутренняя приёмка и подготовка демонстрации | 12-я неделя | Готовая v0.1 и презентационный сценарий |

Срок 90 дней — это нижняя оценка для команды из 4–6 человек при наличии подготовленных пилотных регионов и согласованного списка сценариев.

### 4.2. Состав команды первого этапа

Минимальный состав, достаточный для запуска v0.1 и подготовки к полному MVP ([MVP](MVP.md), раздел «Команда MVP»):

| Роль | Занятость на этапе акселерации |
|------|---------------------------------|
| Руководитель проекта | 1 |
| Архитектор данных/GIS | 1 |
| Data engineer | 1–2 |
| Backend-разработчик | 1 |
| Frontend/GIS-разработчик | 1 |
| Предметный аналитик (экономика, труд, ЧС) | 1 |
| Методолог по работе с данными | 0,5 |

Специалист по информационной безопасности и юрист по данным на этапе v0.1 могут быть привлечены частично — они становятся обязательными при подключении ведомственных контуров.

### 4.3. Параллельные организационные шаги

Технические работы по v0.1 идут параллельно с организационными, обеспечивающими переход к полному MVP:

1. **Поиск государственного оператора пилота.** Это ключевая блокирующая зависимость: без оператора система остаётся в режиме открытых и синтетических данных ([MVP](MVP.md), «Правовая и организационная рамка»; [Заявка на акселерацию](Заявка%20на%20акселерацию.md), раздел 6).
2. **Подготовка шаблонов соглашений.** Шаблоны межведомственных соглашений и «Паспорт набора данных» (см. вопрос 5 [FAQ](Чаво.md)).
3. **Юридическая рамка.** Экспертиза по ФЗ-149, ФЗ-152, ФЗ-187, отраслевым актам и режимам работы с чувствительными слоями.
4. **Запросы в ведомства на доступ.** К Росреестру и НСПД, Минтрансу, Минтруду, МЧС, Росстату и региональным источникам — в разрешённом объёме, в формате «рамка и оператор», а не «дайте данные».
5. **Методические материалы.** Паспорта качества, методика подключения нового слоя, регламент проверки обезличивания (см. вопрос 2.9 [FAQ](Чаво.md)).
6. **Безопасность и аудит.** Описание ролевой модели, журналирования и регламента инцидентов — даже для открытого контура (на случай некорректных публикаций и реидентификации).

### 4.4. Метрики первого этапа

| Метрика | Целевое значение к концу этапа акселерации |
|---------|---------------------------------------------|
| Подключённые открытые наборы | Не менее 12 на пилотный регион |
| Доля наборов с паспортом качества | 100% |
| Реализованные сценарии | Не менее 4 |
| Время сборки регионального паспорта | До 2 минут на запрос |
| Время отклика карты | До 3 секунд для типовых фильтров |
| Доля операций передачи между контурами с записью в журнал | 100% (применимо к API и экспорту отчётов) |
| Подтверждённое обезличивание для открытого контура | 100% |
| Публичная демонстрация | Проведена не менее чем на двух площадках |

### 4.5. Главные риски первого этапа и меры снижения

| Риск | Мера |
|------|------|
| Нет государственного оператора пилота | Платформа остаётся в режиме открытых и синтетических данных; запуск ведомственного контура отложен до появления оператора |
| Сокращение или закрытие части открытых наборов | Каждый набор фиксируется в паспорте качества; при недоступности — переход на резервный источник или синтетику с явной пометкой |
| Соблазн «раздуть» v0.1 до полного MVP | Жёсткое разделение v0.1 и MVP в плане работ и презентациях |
| Несогласованные публикации | Регламент публикации: ни одна выкладка не уходит наружу без проверки на обезличивание |
| Раннее подключение чувствительных слоёв | Запрет на любые чувствительные данные критической инфраструктуры и персональные данные в v0.1 |

## 5. Что получается в итоге

После реализации первых шагов на этапе акселерации появляется:

1. **Платформа v0.1** — публично демонстрируемый прототип открытого контура двойника на 1–2 регионах.
2. **Каталог открытых данных** с паспортами качества по каждому набору.
3. **Методика подключения нового слоя данных**, готовая к использованию ведомственным контуром.
4. **Четыре рабочих сценария**, демонстрирующих практическую пользу: региональный паспорт, кадровый дефицит, сравнение площадок, базовый риск ЧС.
5. **Технологический стек**, переиспользуемый при переходе к полному MVP.
6. **Юридическая и организационная рамка**, готовая к моменту появления государственного оператора пилота.
7. **Список недостающих данных и ведомств**, без которых масштабирование невозможно — это входная точка для следующего этапа.

Этот набор результатов соответствует целевому TRL 4–5 по итогам акселератора и закрывает три исходных вопроса: открытые данные есть, на их основе можно показать работоспособную платформу v0.1, а первые шаги по её реализации уложены в 90-дневную дорожную карту.
