## 1. Цель и границы продукта

### 1.1. Что это

Веб-приложение (SaaS) — **не десктоп, не нативное мобильное** (обоснование — §7.2 [отчёта №1](Отчёт_анализа_2026-05-21_17-52-10_UTC.md)). Доступ через браузер. Сервер контролирует доступ и не выпускает сырые данные пользователю.

### 1.2. Что входит в первую версию (v0.1 → MVP)

| Входит | Не входит (пока) |
|--------|-------------------|
| Открытый контур на 1–2 пилотных регионах | Федеральное покрытие всех субъектов |
| 12 сценариев категории A (см. §4.2 [отчёта №2](Отчёт_анализа_2026-06-01_23-57-56_UTC.md)) | Закрытые/оборонные сценарии |
| Веб-карта + аналитические панели + экспорт | Детальные схемы критической инфраструктуры |
| Каталог данных с паспортами качества | Персональные данные граждан |
| Публичный REST API чтения | Полная автоматизация принятия решений |
| Заготовка ролевой модели и журнала аудита | Замена существующих ГИС (НСПД, ЕМИСС, ГИСП) |

Границы строго наследуют раздел «Что входит / Что не входит в MVP» из [MVP.md](MVP.md).

## 2. Карта функциональных модулей

```
┌─────────────────────────────────────────────────────────────┐
│                     Веб-приложение (SPA)                      │
│  M1 Карта   M2 Сценарии   M3 Каталог данных   M4 Отчёты      │
│  M5 Кабинет/роли   M6 Админка   M7 Журнал аудита            │
└───────────────────────────┬─────────────────────────────────┘
                            │ REST/GraphQL
┌───────────────────────────┴─────────────────────────────────┐
│                    Бэкенд (API + расчёт)                      │
│  S1 Auth/RBAC   S2 Каталог метаданных   S3 Движок сценариев  │
│  S4 Геосервис   S5 Экспорт   S6 Журнал   S7 Контур-шлюз      │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────┴─────────────────────────────────┐
│   ETL/ELT (Airflow) → PostGIS + объектное хранилище          │
│   Каталог метаданных + версии данных/моделей (DVC/MLflow)    │
└──────────────────────────────────────────────────────────────┘
```

### 2.1. Модули фронтенда

| ID | Модуль | Назначение |
|----|--------|------------|
| **M1** | Веб-карта | MapLibre GL JS: слои, фильтры, карточки объектов, тепловые карты дефицитов/рисков, поиск |
| **M2** | Сценарии | Каталог сценариев как пресетов; форма параметров (регион/отрасль/период); запуск; результат |
| **M3** | Каталог данных | Витрина наборов с паспортами качества (источник, дата, лицензия, качество, ограничения) |
| **M4** | Отчёты | Просмотр и экспорт результата в PDF/Markdown; «мои сценарии»; шаринг по ссылке |
| **M5** | Кабинет / роли | Регистрация, профиль, тариф, ключи API, управление подролями (для организаций) |
| **M6** | Админка | Управление слоями, паспортами, тарифами, пользователями (для оператора платформы) |
| **M7** | Журнал аудита | Просмотр действий (для роли оператора/регулятора) |

### 2.2. Сервисы бэкенда

| ID | Сервис | Назначение |
|----|--------|------------|
| **S1** | Auth / RBAC | Аутентификация (Keycloak/ЕСИА-заготовка), роли и права, ключи API, лимиты тарифов |
| **S2** | Каталог метаданных | Реестр слоёв и наборов с паспортами качества и версиями |
| **S3** | Движок сценариев | Именованные пайплайны: тянет слои → прогон через модели → формирует выход; версионирование |
| **S4** | Геосервис | Векторные тайлы, пространственные запросы, агрегация до ОКТМО/района |
| **S5** | Экспорт | Генерация PDF/Markdown-отчётов с источниками, датами, лицензиями, оговорками |
| **S6** | Журнал аудита | Append-only лог: кто/что/когда/в каком контуре/на какой версии данных |
| **S7** | Контур-шлюз (data gateway) | Передача между контурами с проверкой обезличивания и k-анонимности; в v0.1 — только открытый контур |

## 3. Роли и права (RBAC)

На основе типологии пользователей §3.1 [отчёта №1](Отчёт_анализа_2026-05-21_17-52-10_UTC.md). В v0.1 активны только роли открытого контура; остальные — заготовки.

| Роль | Контуры (v0.1) | Ключевые права | Тариф |
|------|----------------|----------------|-------|
| Гость (аноним) | Открытый | Просмотр карты и базовых сценариев | Free |
| Гражданин | Открытый | + сохранение «моих сценариев», экспорт | Free/Pro |
| Бизнес | Открытый | + сценарии 13.x, командное пространство | Pro/Business |
| Разработчик | Открытый | + API-ключи, повышенные лимиты | Free/Enterprise API |
| Исследователь | Открытый, исследовательский (заглушка) | + выгрузка наборов, DOI (позже) | Pro |
| Регион (организация) | Открытый, ведомственный (заглушка) | + white-label, подроли, обучение | Region |
| Оператор инфраструктуры | Открытый, инфраструктурный (заглушка) | + приём «ограничений, не схем» (позже) | по соглашению |
| Оператор платформы (админ) | Все | Управление слоями, тарифами, журналом | — |

**Принципы:** минимально достаточный доступ; нет «суперпользователей» в рабочих сценариях; привилегии временные; все назначения подролей логируются (§4.2 отчёта №1).

## 4. Пользовательские истории и acceptance-критерии (категория A)

Для каждого сценария категории A (§4.2 [отчёта №2](Отчёт_анализа_2026-06-01_23-57-56_UTC.md)) — пользовательская история и критерии приёмки. Ниже — образцы для трёх якорных сценариев; остальные оформляются по тому же шаблону при заведении задач.

### 4.1. Сценарий 1.1 «Региональный паспорт ресурсов» (открытая часть)

> **Как** аналитик региона, **я хочу** выбрать субъект РФ и получить сводный паспорт по открытым слоям, **чтобы** не собирать его вручную неделями.

**Acceptance-критерии:**
- [ ] Пользователь выбирает регион из списка пилотных;
- [ ] Платформа за ≤ 2 минут формирует паспорт: карта + KPI по экономике/занятости/демографии/экологии;
- [ ] Каждый показатель снабжён источником, датой выгрузки, лицензией и оговоркой о неполноте;
- [ ] Результат экспортируется в PDF/Markdown;
- [ ] Запуск записан в журнал аудита (версия данных + версия сценария).

### 4.2. Сценарий 5.1 «Карта кадровых дефицитов»

> **Как** регион/работодатель, **я хочу** видеть, каких специалистов не хватает, **чтобы** планировать обучение и привлечение.

**Acceptance-критерии:**
- [ ] Сопоставление открытых вакансий «Работа России» с агрегированной демографией Росстата и программами Минобрнауки/Минпросвещения;
- [ ] Результат — тепловая карта дефицита по профессиям и муниципалитетам;
- [ ] Фильтры по отрасли/профессии/территории;
- [ ] Указаны источники и дата; видна оговорка «только открытые вакансии».

### 4.3. Сценарий 14.2 «Калькулятор переезда»

> **Как** гражданин, **я хочу** сравнить города по доходам, жилью, инфраструктуре и климату, **чтобы** принять решение о переезде.

**Acceptance-критерии:**
- [ ] Выбор 2–4 городов для сравнения;
- [ ] Сравнительная таблица: агрегированные доходы/жильё (Росстат), инфраструктура (OSM), климат (Росгидромет);
- [ ] Экспорт сравнения; все цифры с источником и датой.

## 5. Модель данных

### 5.1. Унифицированная запись (обязательные атрибуты)

Воспроизводит §5.1 [отчёта №1](Отчёт_анализа_2026-05-21_17-52-10_UTC.md). Любой объект (точка/линия/полигон/статзапись/документ) имеет:

```yaml
record:
  id: str                # устойчивый идентификатор (obj.rosstat.population.2025Q1.16000000)
  source: str            # источник
  source_version: date   # версия/дата выгрузки
  license: str           # условия использования
  quality_flag: enum     # verified | aggregated | draft | outdated
  owner: str             # ответственный за данные
  classifier_codes:      # привязка к классификаторам
    okved: str
    oktmo: str
    kladr: str
  geometry: geojson      # WGS-84 / EPSG:4326 (или ГСК-2011)
  valid_from: date
  valid_to: date
  aggregation_level: enum # region | municipality | settlement | object
  pii_status: enum        # none | aggregated | anonymized | pseudonymized
  contour: enum           # open | departmental | infrastructure | closed | research
```

### 5.2. Паспорт набора данных (dataset passport)

Воспроизводит §6.1 [отчёта №1](Отчёт_анализа_2026-05-21_17-52-10_UTC.md): `id, owner, source, license, update_frequency, classifier_alignment, geometry_crs, pii_status, k_anonymity, known_limitations, validators, signed_by, certificate_version, signed_at`. Хранится в каталоге метаданных (S2), показывается в M3.

### 5.3. Ключевые сущности БД (эскиз)

| Сущность | Назначение |
|----------|------------|
| `datasets` | Наборы данных + паспорта |
| `layers` | Слои карты (ссылаются на datasets) |
| `objects` | Унифицированные записи (PostGIS-геометрия) |
| `scenarios` | Описания сценариев-пресетов + версия пайплайна |
| `scenario_runs` | Запуски: пользователь, параметры, версии данных/модели, результат |
| `users`, `organizations`, `roles`, `api_keys` | Идентичность, RBAC, тарифы |
| `audit_log` | Append-only журнал действий |

### 5.4. Унификация классификаторов

Отрасли → ОКВЭД (маппинг на NACE/ISIC); территория → ОКТМО + КЛАДР/ФИАС + геометрия WGS-84; продукты → ОКПД2 + ТН ВЭД; профессии → ОКЗ; образование → ОКСО; время → ISO 8601 (§5.2 отчёта №1).

## 6. Эскиз API

Базовый принцип: **наружу отдаются разрешённые слои и выводы, не сырые чувствительные записи**; все вызовы журналируются; действуют лимиты тарифа.

### 6.1. REST (v0.1)

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| `GET` | `/api/v1/catalog/datasets` | Список наборов + паспорта (фильтры по домену/региону) |
| `GET` | `/api/v1/catalog/datasets/{id}` | Паспорт конкретного набора |
| `GET` | `/api/v1/layers` | Список доступных слоёв карты |
| `GET` | `/api/v1/tiles/{layer}/{z}/{x}/{y}` | Векторные тайлы слоя |
| `GET` | `/api/v1/objects/{id}` | Карточка объекта |
| `GET` | `/api/v1/scenarios` | Каталог сценариев-пресетов |
| `POST` | `/api/v1/scenarios/{id}/run` | Запуск сценария с параметрами → `run_id` |
| `GET` | `/api/v1/runs/{run_id}` | Статус и результат запуска |
| `GET` | `/api/v1/runs/{run_id}/export?format=pdf\|md` | Экспорт отчёта |
| `POST` | `/api/v1/auth/keys` | Создать API-ключ (по тарифу) |

GraphQL — опционально на этапе MVP для гибких выборок каталога.

### 6.2. Контракты ответов

Каждый ответ с данными включает блок происхождения: `source, source_version, license, quality_flag, known_limitations` — это требование доверия (M3/S2) и обязательное условие открытого контура.

## 7. Нефункциональные требования

| Категория | Требование (цель) |
|-----------|--------------------|
| **Производительность** | Время отклика карты ≤ 3 с для типовых фильтров; сборка регионального паспорта ≤ 2 мин |
| **Масштабируемость** | Горизонтальное масштабирование API; тайлы через CDN |
| **Безопасность** | Ролевой доступ, минимально достаточные права, шифрование транзита, ключи API с лимитами |
| **Обезличивание** | 100% наборов открытого контура с подтверждённым обезличиванием; k-анонимность; квартальная проверка реидентификации |
| **Наблюдаемость** | Журнал аудита (append-only, по возможности с цепочкой целостности); метрики; алёрты |
| **Воспроизводимость** | Каждый запуск фиксирует `dataset_version + model_version + scenario_version`; результат без версий отбраковывается |
| **Доступность** | SLA 99,5% для платных планов; деградация без потери журнала |
| **Импортозамещение** | Приоритет отечественных компонентов; план замены (сценарий 15.8) |
| **Доступная среда** | Адаптивный дизайн; учёт требований доступности интерфейса |

## 8. Технологический стек

| Метаданные/трансформации | dbt |
| Аутентификация | Keycloak (или отечественный аналог) + интеграция с ЕСИА |
| Фронтенд | React или Vue |
| Карта | MapLibre GL JS; 3D — CesiumJS |
| Модели | DoWhy/EconML/CausalML, NetworkX, SimPy, mesa, OR-Tools, Pyomo |
| Версии моделей/данных | MLflow + DVC |
| Журнал | Append-only log + ELK или ClickHouse |
