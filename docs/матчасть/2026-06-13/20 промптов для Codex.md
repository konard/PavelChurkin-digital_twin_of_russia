# 20 промптов для Codex: разработка SaaS-приложения «Цифровой двойник России»

> **Дата:** 2026-06-09
> **Документ-ответ на:** [issue #40](https://github.com/PavelChurkin/resource-based-economy-Article/issues/40)
> **Связанный PR:** [#41](https://github.com/PavelChurkin/resource-based-economy-Article/pull/41)
> **Назначение:** пошаговый план разработки функционала и функциональных модулей SaaS-веб-приложения, разбитый на **20 промптов для Codex** (или другого кодового агента)

Документ отвечает на требование [issue #40](https://github.com/PavelChurkin/resource-based-economy-Article/issues/40): *«Разбей разработку функционала и функциональных модулей SaaS веб-приложения на 20 промптов для Codex»*. Он превращает функциональную спецификацию ([Функционал SaaS](Функционал%20SaaS.md)) в **исполнимую последовательность задач**.

## Как пользоваться

- **Очередность.** Промпты идут в порядке зависимости: каждый следующий опирается на результат предыдущих. Можно распараллелить промпты внутри одного блока, но не перескакивать через блок.
- **Контур.** Все промпты относятся **исключительно к открытому контуру** (требование issue): без ведомственных, инфраструктурных и закрытых слоёв.
- **Источники данных.** ETL-промпты (4–7) подключают источники в порядке **«сначала CSV, потом API/JSON»** из [Источники CSV и API](Источники%20CSV%20и%20API.md).
- **Модули.** Идентификаторы модулей (M1–M7) и сервисов (S1–S7) — из [Функционал SaaS](Функционал%20SaaS.md), §2.
- **Формат промпта.** Каждый промпт самодостаточен: цель, что создать, критерии готовности. Перед запуском подставьте имя пилотного региона и репозитория.

Карта промптов по блокам:

| Блок | Промпты | Результат блока |
|------|---------|------------------|
| A. Каркас и инфраструктура | 1–3 | Репозиторий, инфраструктура, БД-схема |
| B. Данные (ETL: CSV → API) | 4–7 | Загруженные слои с паспортами |
| C. Бэкенд (API + сценарии) | 8–12 | REST API, геосервис, движок сценариев, экспорт, журнал |
| D. Фронтенд (модули M1–M7) | 13–17 | Карта, сценарии, каталог, отчёты, кабинет |
| E. Качество и выпуск | 18–20 | Тесты, наблюдаемость, демо-режим и деплой |

## Блок A. Каркас проекта и инфраструктура

### Промпт 1 — Инициализация монорепозитория

```
Создай структуру монорепозитория digital-twin-platform по разделу 9
документа «Функционал SaaS»: каталоги backend/ (app, etl, models, tests),
frontend/ (src, tests), infra/, data/, docs/, .github/workflows/.
Добавь README.md с описанием проекта и архитектуры, LICENSE, .gitignore,
.editorconfig. Backend — Python 3.12 + FastAPI (pyproject.toml, зависимости).
Frontend — Vite + React + TypeScript. Настрой pre-commit (ruff, black, eslint,
prettier). Критерий готовности: репозиторий собирается, линтеры проходят,
README описывает запуск.
```

### Промпт 2 — Инфраструктура локальной разработки

```
Создай infra/docker-compose.yml со службами: PostgreSQL 16 + PostGIS,
объектное хранилище (MinIO, S3-совместимое), Redis (очереди), Apache Airflow
(scheduler + webserver), сервис backend и frontend. Добавь Makefile с командами
up/down/migrate/seed/test. Опиши переменные окружения в .env.example.
Критерий готовности: `make up` поднимает весь стек, healthcheck всех служб
зелёный, PostGIS-расширение включено.
```

### Промпт 3 — Схема базы данных и миграции

```
Реализуй модель данных из раздела 5 «Функционал SaaS» через SQLAlchemy +
Alembic. Таблицы: datasets (с полями паспорта качества), layers, objects
(PostGIS-геометрия, EPSG:4326), scenarios, scenario_runs (с полями
dataset_version, model_version, scenario_version), users, organizations,
roles, api_keys, audit_log (append-only). У objects — обязательные атрибуты
унифицированной записи (source, source_version, license, quality_flag,
classifier_codes, aggregation_level, pii_status, contour). Создай первую
миграцию и сидер справочников (ОКТМО/ОКВЭД2/ОКЗ). Критерий готовности:
`alembic upgrade head` проходит, сид загружается, есть тест на создание объекта.
```

## Блок B. Данные (ETL по принципу «сначала CSV, потом API/JSON»)

### Промпт 4 — Базовый ETL-фреймворк и паспорт качества

```
Создай в backend/etl каркас ETL: базовый класс Source (методы fetch, parse,
normalize, load) и обязательное формирование «паспорта качества» для каждого
набора (источник, дата выгрузки = source_version, лицензия, периодичность,
known_limitations, pii_status). Поддержи кодировки Windows-1251/UTF-8,
форматы CSV/XLSX/XML. Регистрируй каждый набор в таблице datasets.
Добавь Airflow DAG-шаблон с расписанием. Критерий готовности: тест прогоняет
синтетический CSV через пайплайн и создаёт запись datasets с заполненным паспортом.
```

### Промпт 5 — Коннекторы очереди 1 (CSV/файлы) — геооснова и статистика

```
Реализуй коннекторы очереди 1 из документа «Источники CSV и API», §2 и §5:
(1) OSM Geofabrik (.osm.pbf пилотного региона → дороги, POI, границы в PostGIS);
(2) Росстат opendata (CSV/XLSX, экономика/демография); (3) ЕМИСС fedstat (XLSX);
(4) ГАР/ФИАС (адреса пилотного региона). Все слои нормализуются к
унифицированной записи и привязываются к ОКТМО. Критерий готовности: для
пилотного региона в БД появляются слои с паспортами; есть тесты парсинга.
```

### Промпт 6 — Коннекторы очереди 1 — труд, образование, бюджет, климат

```
Добавь коннекторы очереди 1: «Работа России» (выгрузки CSV), Минпросвещения,
Минобрнауки (CSV), ФНС opendata (CSV), zakupki.gov.ru и spending.gov.ru
(XML/CSV дампы), Росгидромет meteo.ru (климатические ряды CSV). Каждый — с
паспортом качества и расписанием обновления (квартал/год по §2 «Источники
CSV и API»). Критерий готовности: 12 наборов минимального стартового набора
(§5 «Источники CSV и API») загружаются одной командой `make seed-open-data`.
```

### Промпт 7 — Коннекторы очереди 2 (API/JSON) для свежих данных

```
Подключи источники очереди 2 из «Источники CSV и API», §3: «Работа России» API
(JSON, ежедневно), NASA FIRMS (JSON/CSV, очаги пожаров, до 4× в сутки, ключ
MAP_KEY из env). Реализуй инкрементальную загрузку, обработку пагинации и
лимитов, ретраи. Привяжи к расписанию Airflow. Критерий готовности: коннекторы
обновляют слои по расписанию, при недоступности API падают мягко (last-known +
пометка outdated в quality_flag), есть тесты с мок-ответами.
```

## Блок C. Бэкенд: API, геосервис, сценарии, экспорт, журнал

### Промпт 8 — Каталог данных и API метаданных (S2)

```
Реализуй сервис каталога метаданных (S2) и REST-эндпоинты из «Функционал SaaS»,
§6.1: GET /api/v1/catalog/datasets (фильтры по домену/региону),
GET /api/v1/catalog/datasets/{id}. Каждый ответ с данными содержит блок
происхождения (source, source_version, license, quality_flag,
known_limitations). Критерий готовности: эндпоинты возвращают паспорта
загруженных наборов; OpenAPI-схема сгенерирована; есть контрактные тесты.
```

### Промпт 9 — Геосервис и векторные тайлы (S4)

```
Реализуй геосервис (S4): GET /api/v1/layers, GET /api/v1/tiles/{layer}/{z}/{x}/{y}
(векторные тайлы MVT из PostGIS), GET /api/v1/objects/{id} (карточка объекта),
пространственная агрегация до ОКТМО/района. Кэширование тайлов. Критерий
готовности: тайлы отдаются за ≤3 c для типовых запросов; карточка объекта
содержит блок происхождения; есть тест на агрегацию.
```

### Промпт 10 — Движок сценариев (S3) и 3 якорных сценария категории A

```
Реализуй движок сценариев (S3): именованные пайплайны «тянет слои → прогон →
выход», с версионированием (scenario_version, model_version). Эндпоинты:
GET /api/v1/scenarios, POST /api/v1/scenarios/{id}/run → run_id,
GET /api/v1/runs/{run_id}. Реализуй 3 якорных сценария из «Функционал SaaS», §4:
1.1 «Региональный паспорт», 5.1 «Карта кадровых дефицитов», 14.2 «Калькулятор
переезда». Каждый запуск пишет в scenario_runs dataset_version + model_version +
scenario_version. Критерий готовности: региональный паспорт собирается ≤2 мин;
результат воспроизводим при тех же версиях.
```

### Промпт 11 — Экспорт отчётов (S5)

```
Реализуй сервис экспорта (S5): GET /api/v1/runs/{run_id}/export?format=pdf|md.
Отчёт обязательно включает по каждому показателю источник, дату выгрузки,
лицензию и оговорку о неполноте; в подвал — использованные версии
(dataset/model/scenario). Критерий готовности: PDF и Markdown генерируются,
проходят проверку наличия блока происхождения; есть golden-file тест.
```

### Промпт 12 — Журнал аудита (S6) и контур-шлюз (S7)

```
Реализуй журнал аудита (S6) как append-only лог: кто/что/когда/в каком контуре/
на какой версии данных, с цепочкой целостности (hash chain). Реализуй
контур-шлюз (S7): в v0.1 разрешён только открытый контур; любая передача
проверяется на обезличивание/k-анонимность и логируется. Критерий готовности:
100% запусков сценариев и экспортов попадают в журнал; есть тест на
неизменяемость лога и блокировку неоткрытых контуров.
```

## Блок D. Фронтенд: модули M1–M7

### Промпт 13 — Веб-карта (M1)

```
Реализуй модуль карты (M1) на MapLibre GL JS: подключение векторных тайлов
(S4), переключение слоёв, фильтры, карточки объектов, тепловые карты
дефицитов/рисков, поиск по адресу (ГАР). Дизайн адаптивный, учёт доступной
среды. Критерий готовности: карта пилотного региона рендерится, фильтры
работают, карточка объекта показывает блок происхождения.
```

### Промпт 14 — Каталог данных (M3)

```
Реализуй модуль каталога данных (M3): витрина наборов с паспортами качества
(источник, дата, лицензия, качество, ограничения), фильтры по домену/региону,
карточка набора. Данные — из API каталога (промпт 8). Критерий готовности:
все загруженные наборы видны с полными паспортами; пустые/устаревшие наборы
помечены визуально.
```

### Промпт 15 — Сценарии и запуск (M2)

```
Реализуй модуль сценариев (M2): каталог сценариев-пресетов, форма параметров
(регион/отрасль/период), запуск (POST .../run), индикатор прогресса, переход к
результату. Свяжи с движком сценариев (промпт 10). Критерий готовности:
пользователь запускает «Региональный паспорт» и «Карту кадровых дефицитов» и
получает результат с источниками и оговорками.
```

### Промпт 16 — Отчёты и шаринг (M4)

```
Реализуй модуль отчётов (M4): просмотр результата запуска, экспорт в PDF/Markdown
(S5), «мои сценарии», шаринг по ссылке. Каждый результат показывает версии
данных/модели/сценария. Критерий готовности: результат экспортируется,
ссылка-шаринг открывает тот же воспроизводимый результат.
```

### Промпт 17 — Кабинет, роли и демо/гостевой режим (M5 + S1)

```
Реализуй кабинет (M5) и заготовку Auth/RBAC (S1): регистрация/вход, профиль,
роли открытого контура (Гость, Гражданин, Бизнес, Разработчик), API-ключи,
лимиты по тарифам. Реализуй ГОСТЕВОЙ/ДЕМО-режим (см. отчёт №3, раздел о госте):
неавторизованный гость работает с демо-сценариями ТОЛЬКО ДЛЯ ЧТЕНИЯ — он не
сохраняет «мои сценарии», не получает API-ключи и не запускает обновление
данных; ему доступны предрассчитанные результаты и просмотр карты/каталога.
Критерий готовности: гость видит демо, но действия записи заблокированы и
предлагают регистрацию; зарегистрированный пользователь получает полный
открытый контур.
```

## Блок E. Качество, наблюдаемость и выпуск

### Промпт 18 — Тесты и CI

```
Покрой ключевые пути тестами: ETL (парсинг форматов/кодировок), API (контрактные
+ интеграционные), движок сценариев (воспроизводимость при фиксированных
версиях), фронтенд (компонентные + e2e на Playwright для трёх якорных
сценариев). Настрой .github/workflows: линт, типы, тесты, сборка, прогон на PR.
Критерий готовности: CI зелёный, покрытие критических модулей ≥70%,
e2e проходит для гостевого и авторизованного сценариев.
```

### Промпт 19 — Нефункциональные требования и наблюдаемость

```
Внедри нефункциональные требования из «Функционал SaaS», §7: метрики
(Prometheus), структурные логи, алёрты, лимиты частоты на API, шифрование
транзита, заголовки безопасности. Проверь цели производительности (карта ≤3 c,
паспорт ≤2 мин). Добавь регламент обезличивания/k-анонимности для открытого
контура и квартальную проверку реидентификации (хотя бы скриптом). Критерий
готовности: дашборд метрик работает, нагрузочный тест подтверждает цели,
проверка обезличивания запускается командой.
```

### Промпт 20 — Демо-стенд, сидер и деплой

```
Подготовь публичный демо-стенд v0.1: сид одного пилотного региона (12 наборов
очереди 1 + 2 коннектора очереди 2), 4 демонстрационных сценария (региональный
паспорт, кадровый дефицит, сравнение площадок, базовый риск ЧС), гостевой
демо-режим. Напиши инфраструктуру деплоя (docker-compose/IaC), инструкцию
запуска и страницу «Об источниках и ограничениях». Критерий готовности:
по инструкции стенд поднимается с нуля, демо-сценарии открываются гостем,
каждая панель показывает источники/даты/лицензии/оговорки.
```
