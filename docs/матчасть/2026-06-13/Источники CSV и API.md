# Источники открытых данных: CSV-выгрузки и API «Цифрового двойника России»

> **Дата:** 2026-06-09
> **Документ-ответ на:** [issue #40](https://github.com/PavelChurkin/resource-based-economy-Article/issues/40)
> **Связанный PR:** [#41](https://github.com/PavelChurkin/resource-based-economy-Article/pull/41)
> **Назначение:** рабочий каталог ссылок на источники открытых данных для ETL открытого контура, отсортированный по принципу **«сначала CSV — потом API/JSON»**

Документ отвечает на требование [issue #40](https://github.com/PavelChurkin/resource-based-economy-Article/issues/40): *«составь список ссылок на эти сервисы и скачивание CSV»*. Он развивает каталог из [Открытые данные](Открытые%20данные.md), §2, доводя его до уровня **конкретных точек загрузки** и **формы обновления базы данных**.

Все ссылки даны по состоянию на **июнь 2026 года** и подлежат повторной проверке перед использованием: состав открытых наборов в РФ заметно менялся в 2023–2025 годах (см. [Открытые данные](Открытые%20данные.md), §2.8). Источники относятся **только к открытому контуру**; чувствительные сетевые и персональные данные сюда не входят принципиально ([FAQ](Чаво.md), вопрос 6).

## 1. Принцип отбора: сначала CSV, потом API

[Issue #40](https://github.com/PavelChurkin/resource-based-economy-Article/issues/40) задаёт порядок подключения источников:

> *«…из тех сервисов, что предлагают CSV, только потом на API с json, чтобы можно было с определённой регулярностью обновлять базу данных».*

Это разумная инженерная очередность, и она зафиксирована как правило ETL открытого контура:

| Очередь | Форма доступа | Почему так | Регулярность обновления |
|---------|---------------|------------|--------------------------|
| **Очередь 1 (сейчас)** | **CSV / XLSX / дамп-файл** по прямой ссылке | Не требует ключей, согласований и обработки лимитов; ETL — это «скачать файл → распарсить → загрузить» | Ручной/по расписанию: раз в квартал/месяц, по факту обновления файла на стороне ведомства |
| **Очередь 2 (потом)** | **REST API с JSON** | Даёт инкрементальные обновления и более свежие данные, но требует ключей, обработки пагинации и лимитов | Автоматический коннектор по расписанию (Airflow): ежедневно/еженедельно |
| **Очередь 3 (по необходимости)** | **Геодампы** (`.osm.pbf`, Shapefile, GeoJSON) | Большие пространственные слои; обновляются редко, но тяжёлые | По расписанию: раз в месяц/квартал |

**Практическое следствие для v0.1:** ETL-пайплайн стартует на источниках **очереди 1** (CSV-файлы), чтобы за недели получить рабочую базу без единого согласования. Источники **очереди 2** (API/JSON) подключаются вторым шагом для тех доменов, где важна свежесть (вакансии, пожары, котировки воздуха). Это полностью совпадает с архитектурой v0.1 ([Открытые данные](Открытые%20данные.md), §3.3) и с движком ETL ([Функционал SaaS](Функционал%20SaaS.md), §8).

Формула обновления базы данных:

```
[Источник CSV] --(Airflow DAG по расписанию)--> [Загрузка файла]
   --> [Нормализация + паспорт качества] --> [PostGIS / объектное хранилище]
   --> [Каталог метаданных: source_version = дата выгрузки]
```

Для каждого источника ниже указаны: **домен**, **что даёт**, **прямая точка доступа**, **формат**, **очередь подключения** и **рекомендованная периодичность обновления**.

## 5. Минимальный стартовый набор для ETL v0.1

Для запуска ETL открытого контура **прямо сейчас** достаточно 12 наборов из очереди 1 + 2 коннекторов очереди 2. Это и есть рекомендованный «день первый» для код-команды:

```
Очередь 1 (CSV/файлы, без ключей) — грузим первыми:
  1. OSM Geofabrik (пилотный регион)         .osm.pbf
  2. Росстат opendata (экономика/демография)  CSV/XLSX
  3. ЕМИСС fedstat (межвед. показатели)       XLSX
  4. ФНС opendata (численность/спецрежимы)    CSV
  5. «Работа России» opendata (вакансии)      CSV
  6. Минпросвещения opendata (школы)          CSV
  7. Минобрнауки opendata (вузы/программы)    CSV
  8. ГАР/ФИАС (адреса пилотного региона)      XML/DBF
  9. Росгидромет meteo.ru (климат)            CSV
 10. zakupki.gov.ru (контракты территории)    XML/CSV
 11. spending.gov.ru (расходы)                CSV
 12. Классификаторы ОКТМО/ОКВЭД2/ОКЗ          CSV

Очередь 2 (API/JSON) — подключаем вторыми ради свежести:
  A. «Работа России» API (вакансии, сутки)    JSON
  B. NASA FIRMS (пожары, до 4× в сутки)        JSON/CSV
```

Каждый загруженный набор получает **паспорт качества** (источник, дата выгрузки = `source_version`, лицензия, периодичность, ограничения) — обязательное условие открытого контура ([Функционал SaaS](Функционал%20SaaS.md), §5.2; [отчёт №1](Отчёт_анализа_2026-05-21_17-52-10_UTC.md), §6.1).

## 6. Оговорки и правила работы с источниками

1. **Проверка перед использованием.** Состав и адреса наборов меняются; перед каждым релизом ETL ссылки проверяются заново.
2. **Кодировки и форматы.** Преобладают CSV в Windows-1251, XLSX, XML; геоданные — Shapefile/MapInfo. ETL обязан корректно их обрабатывать ([Открытые данные](Открытые%20данные.md), §2.8).
3. **Лимиты массового доступа.** ЕГРЮЛ, ГИР БО ограничивают массовую выгрузку — для открытого контура это не критично (работаем с агрегатами).
4. **Лицензии.** Для каждого набора фиксируется лицензия; OSM — ODbL (требует указания авторства и share-alike для производных).
5. **Только открытый контур.** Сетевые схемы, режимы инфраструктуры и персональные данные сюда не входят — это предмет инфраструктурного и закрытого контуров ([FAQ](Чаво.md), вопрос 6; [Шаблон соглашения с оператором инфраструктуры](Шаблон%20соглашения%20с%20оператором%20инфраструктуры.md)).
