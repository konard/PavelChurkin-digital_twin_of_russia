# Презентация «Цифрового двойника России»

## Назначение документа

Этот документ — презентационная сборка по проекту «Цифровой двойник России». Он отвечает на три вопроса, поднятых в обсуждении:

1. Как устроен цифровой двойник как система — принципиальная и функциональная схема, общая структура.
2. Какие данные он в принципе может обрабатывать, какие обрабатывает на каждом контуре, а какие — не обрабатывает (сводная таблица).
3. Что даёт цифровой двойник применительно к потребностям из [Тезисов](../Тезисы.md): какие пункты являются основными, где технологическая реализация пока неочевидна и какие шаги могут быть предприняты для её прояснения.

Документ опирается на уже подготовленные материалы и не дублирует их в полном объёме:

- [Описание](../Описание.md) — концепция, аналоги, состав услуг.
- [MVP](../MVP.md) — границы минимально жизнеспособной версии, технологический стек, этапы.
- [Тезисы](../Тезисы.md) — ключевые тезисы и расширенный раздел потребностей.
- [Примеры обработки информации для контуров](../Примеры%20обработки%20информации%20для%20контуров.md) — сценарии и правила передачи данных между контурами.

## 1. Принципиальная схема

> **Цифровой двойник России — это контур принятия решений, а не публичная карта.** На принципиальном уровне он связывает источники данных, защищённую обработку и пользователей разных уровней через систему из пяти контуров доступа.

### 1.1. Принципиальная схема (общий поток)

```
┌──────────────────────────────────────────────────────────────────────┐
│                       ИСТОЧНИКИ ДАННЫХ                               │
│  НСПД · ЕМИСС · ФГИС ЕФГИ · отраслевые системы · региональные ИС     │
```

Главный принцип — однонаправленное обогащение: чем выше чувствительность данных, тем меньше деталей выходит наружу. Между контурами передаются не «сырые данные», а решения, ограничения и обезличенные выводы (правила передачи зафиксированы в [Примеры обработки информации для контуров](../Примеры%20обработки%20информации%20для%20контуров.md)).

## 2. Функциональная схема

> **Функциональная схема описывает, какие функции система выполняет, как они связаны и какой результат получает пользователь.** Этот раздел уточняет «что система делает» в дополнение к «как она устроена».

### 2.1. Функциональные блоки

| № | Функциональный блок | Что делает | Где живёт результат |
|---|---------------------|------------|---------------------|
| Ф1 | Подключение источников | Договоры, регламенты, технические коннекторы к ведомственным и отраслевым системам | Каталог источников и их паспортов |
| Ф2 | Приём и нормализация | ETL/ELT, валидация, геокодирование, приведение классификаторов | Слой нормализованных данных |
| Ф3 | Каталог и паспортизация | Метаданные, происхождение, лицензии, качество, владельцы | Каталог метаданных |
| Ф4 | Хранение и версионирование | Пространственная БД, объектное хранилище, версии моделей и данных | Хранилище со снимками версий |
| Ф5 | Аналитика и сценарии | Расчёт показателей, поиск дефицитов, моделирование «что если» | Сервис сценариев и аналитики |
| Ф6 | Визуализация | Карта/3D, аналитические панели, отчёты, экспорт | Веб-интерфейсы по контурам |
| Ф7 | Доступ и роли | Аутентификация, ролевой доступ, контуры, журнал аудита | Сервис управления доступом |
| Ф8 | API и публикация | Открытые и регламентированные API, ленты данных | Каталог API |
| Ф9 | Управление качеством | Паспорта, проверки, обратная связь о неточностях | Регламенты качества |
| Ф10 | Безопасность и аудит | Журналирование, контроль раскрытия, обезличивание | Сервис аудита |

### 2.2. Связи между блоками

```
[Ф1 Источники] ──► [Ф2 Нормализация] ──► [Ф3 Каталог] ──► [Ф4 Хранение]
                                                              │
                                                              ▼
                                            ┌───── [Ф5 Аналитика и сценарии] ─────┐
                                            │                                     │
                                            ▼                                     ▼
                                  [Ф6 Визуализация]                       [Ф8 API и публикация]
                                            │                                     │
                                            └────────────┬────────────────────────┘
                                                         ▼
                                              [Ф7 Доступ и роли] ◄── [Ф10 Безопасность и аудит]
                                                         │
                                                         ▼
                                              [Контуры пользователей]

[Ф9 Управление качеством] подключается ко всем блокам как сквозной процесс.
```

### 2.3. Что система делает на каждом контуре (функциональный срез)

| Функция | Открытый | Ведомственный | Инфраструктурный | Закрытый | Исследовательский |
|---------|:--------:|:-------------:|:----------------:|:--------:|:-----------------:|
| Просмотр карт и панелей | агрегаты | детализация в полномочиях | только релевантный участок | только утверждённый круг | синтетика и обезличенные |
| Поиск объектов | по обезличенным | по реестрам пилота | по сетям и узлам | ограниченный, журнал | по тестовым наборам |
| Расчёт показателей | публичные | межведомственные | устойчивость и зависимости | специальные сценарии | экспериментальные |
| Сценарное моделирование | нет | да, в полномочиях | да, по объекту | да, ограниченный круг | да, на синтетике |
| Загрузка новых наборов | нет | по согласованию | от оператора инфраструктуры | от уполномоченного органа | от научной команды |
| Экспорт и отчёты | публичные | регламентированный | без раскрытия схем | строго ограниченный | методические материалы |
| API | открытое | регламентированное | по согласованию с оператором | нет внешнего | для образования и науки |

## 3. Структура цифрового двойника

### 3.1. Слои данных (предметная структура)

```
   Слой ресурсов          Слой производства         Слой логистики
   (недра, вода, лес,     (мощности, цепочки,      (дороги, узлы,
    земли, климат)         поставщики, продукция)   маршруты, склады)
        │                       │                         │
        └───────────┬───────────┴───────────┬─────────────┘
                    ▼                       ▼
            Слой кадров            Слой инфраструктуры
            (вакансии, обуче-      (энергетика, связь,
             ние, миграция,         вода, ЖКХ, транспорт)
             компетенции)                 │
                    │                     │
                    └─────────┬───────────┘
                              ▼
                       Слой рисков
                       (ЧС, экология, кли-
                        матические сцена-
                        рии, точки отказа)
                              │
                              ▼
                     Слой сценариев и решений
                     (план/факт, «как есть»,
                      «после проекта», резерв)
```

Структура отражает связи: ни один слой не существует изолированно. Решение по производству изменяет нагрузку на инфраструктуру и кадры, а изменение климатического слоя меняет риски логистики и устойчивости сетей.

### 3.2. Уровни управления

| Уровень | Что отображает | Типовые пользователи |
|---------|----------------|----------------------|
| Федеральный | Межрегиональные потоки, общенациональные программы, баланс ресурсов | Федеральные органы, координационные центры |
| Региональный | Региональный паспорт, дефициты, инвестиции, кадры | Правительство региона, отраслевые министерства |
| Муниципальный | Локальные ресурсы, инфраструктура, образование, занятость | Муниципалитеты, местные операторы |
| Отраслевой | Производственные кластеры, отраслевая логистика, кадры | Отраслевые ассоциации, операторы кластеров |
| Объектный | Конкретные объекты, площадки, узлы | Операторы, инвесторы, эксплуатация |

### 3.3. Технологический стек (целевой контур, опираясь на [MVP](../MVP.md))

| Слой | Компоненты | Назначение |
|------|------------|------------|
| Хранение | PostgreSQL/PostGIS, объектное хранилище | Пространственные данные, архивы, снимки |
| Обработка | Python/FastAPI, Apache Airflow или аналог | API, расчёт сценариев, ETL/ELT |
| Визуализация | React/Vue, MapLibre/CesiumJS | Карта, 3D, панели, отчёты |
| Доступ | Keycloak или отечественный аналог, контуры доступа | Роли, аутентификация, разграничение |
| Журналирование | Журнал аудита, версионирование наборов и моделей | Подотчётность, расследование инцидентов |
| Каталог | Каталог метаданных, паспорта качества | Прослеживаемость и доверие к данным |

Принцип импортозамещения и суверенитета (см. [Тезис 10](../Тезисы.md#тезис-10-технологическая-независимость)) — приоритет отечественного стека и открытых стандартов, переносимость компонентов, отказ от зависимости от единственного поставщика на каждом критическом уровне.

## 4. Сводная таблица: какие данные обрабатывает каждый контур

> **Принцип:** один и тот же класс данных может попасть в разные контуры, но в разной детализации и с разной защитой. Здесь сведено, что в принципе может обрабатывать система, что попадает на каждый контур и что не должно попадать ни в один контур.

### 4.1. Что цифровой двойник в принципе может обрабатывать

| Класс данных | Содержание | Источники |
|--------------|------------|-----------|
| Пространственные | Границы, кадастр, инфраструктура, рельеф, гидрография | НСПД, OSM, региональные ГИС |
| Статистические | Демография, занятость, отрасли, экономика | ЕМИСС/Росстат |
| Ресурсные | Недра, вода, лес, земли, климат | ФГИС ЕФГИ, Роснедра, Росгидромет |
| Производственные | Мощности, цепочки, продукция, поставщики | Отраслевые системы, реестры предприятий |
| Логистические | Транспорт, узлы, маршруты, складские мощности | ОАО «РЖД», Минтранс, операторы |
| Кадровые | Вакансии, компетенции, образование | Службы занятости, Минобрнауки |
| Инфраструктурные | Энергетика, связь, вода, ЖКХ | Сетевые операторы, регуляторы |
| Экологические | Воздух, вода, отходы, охранные зоны | Росприроднадзор, Росгидромет |
| Рисковые | ЧС, паводки, пожары, аварии | МЧС, региональные службы |
| Климатические | Долгосрочные сценарии, прогнозы | Росгидромет, научные центры |
| Сценарные | «Что если» по проектам и программам | Внутренние модели системы |
| Метаданные | Паспорта качества, происхождение, лицензии, версии | Внутренний каталог |

### 4.2. Что обрабатывается на каждом контуре, а что — нет

Сводная таблица: для каждого класса данных указано, что попадает в контур, а что — нет. Колонка «Не обрабатывается ни одним контуром» собирает то, что система намеренно не принимает.

| Класс данных | Открытый | Ведомственный | Инфраструктурный | Закрытый | Исследовательский | Не обрабатывается ни одним контуром |
|--------------|----------|---------------|------------------|----------|-------------------|-------------------------------------|
| Пространственные | Агрегированные карты, границы, базовая инфраструктура | Кадастровые слои, объекты пилота | Сети и узлы операторов | Чувствительные объекты в обобщённом виде | Тестовые наборы и синтетика | Точные координаты специальных объектов и закрытых режимных территорий |
| Статистические | Открытая статистика и индексы | Детализация по муниципалитетам и отраслям | Не профиль контура | Стат. срезы по чувствительным сегментам в обобщённом виде | Исторические выгрузки для моделей | Персональные сведения о гражданах |
| Ресурсные | Агрегированный потенциал территории | Месторождения, переработка, потребители (в полномочиях) | Доступы, подъезды, ограничения по сети | Резервы и стратегические запасы | Модельные баланс-расчёты на синтетике | Точные координаты чувствительных запасов и их объёмы |
| Производственные | Факт проекта, агрегированные рабочие места | Мощности, потребности, поставщики, бюджеты пилота | Подключения и условия | Чувствительные производственные планы и зависимости | Модели прогноза дефицитов | Непубличные коммерческие тайны без согласия владельца |
| Логистические | Транспортные коридоры, типовые задержки | Альтернативные маршруты грузопотоков | Пропускная способность узлов и зависимости | Специальные/кризисные маршруты | Модели резервирования на синтетике | Расписания защищённых перевозок и точные специальные маршруты |
| Кадровые | Вакансии и программы обучения (агрегированно) | Связка работодателей, программ, мер поддержки | Не профиль контура | Не профиль контура (передаются обобщённые потребности) | Прогнозные модели на обезличенных данных | Персональные данные работников и их контакты |
| Инфраструктурные | Обобщённый статус «можно/нельзя/при условиях» | Выводы для согласования проектов | Схемы сетей, режимы, зависимости | Чувствительные точки отказа в обобщённом виде | Модели устойчивости на тестовых сетях | Полные схемы сетей и режимы работы за пределами инфраструктурного контура |
| Экологические | Агрегированные показатели качества среды | Нагрузки и ограничения по проектам | Уязвимые объекты в зоне риска | Чувствительные сценарии экокризисов | Модели на исторических данных | Точечные данные с датчиков, по которым возможна реидентификация владельца |
| Рисковые / ЧС | Зоны паводков, пожаров (обобщённо) | Ресурсы реагирования, план/факт | Уязвимые объекты и зависимости | Сценарии специальных служб | Модели прогноза рисков | Точные точки и режимы работы специальных служб |
| Климатические | Сценарии и прогнозы (обобщённо) | Региональные адаптационные планы | Воздействие на критическую инфраструктуру | Чувствительные адаптационные сценарии | Климатические модели | Сырые непроверенные прогнозы без паспорта качества |
| Сценарные | Не обрабатываются (только публикация результатов) | Базовые сценарии запуска проектов | Сценарии устойчивости сетей | Кризисные и специальные сценарии | Эксперименты на синтетике | Сценарии без указания ограничений и допущений |
| Метаданные | Открытые паспорта качества и описания API | Полные паспорта в полномочиях | Метаданные сетевых слоёв | Метаданные ограниченного доступа | Описания экспериментов | Метаданные без указания происхождения и владельца |

Жёсткие границы (одинаковы для всех контуров):

- **Не обрабатываются нигде:** персональные данные граждан без правового основания и согласия; точные координаты критической инфраструктуры в открытом виде; идентификаторы микропредприятий, по которым возможна реидентификация; сведения, отнесённые к государственной тайне, без специального режима хранения; данные без паспорта качества.
- **Не передаются между контурами «как есть»:** между контурами идут не сырые наборы, а решения, ограничения и обезличенные выводы. Это правило закреплено в [Примеры обработки информации для контуров](../Примеры%20обработки%20информации%20для%20контуров.md).

## 5. Что даёт цифровой двойник: основное из «потребностей» и шаги по технологической реализации

> Раздел отвечает на вопрос: **исходя из тезисов о потребностях, что даёт цифровой двойник как основное и где технологическая реализация пока неочевидна — и какие шаги можно предпринять.**

Источник анализа — [Тезис 2 «Потребности, которые решает проект»](../Тезисы.md#тезис-2-потребности-которые-решает-проект-расширенно), где выделено семь доменов потребностей и подробная сводка симптомов и последствий.

### 5.1. Основное, что даёт цифровой двойник

Из расширенного раздела потребностей выделены пять «главных продуктов» системы — то, что сильнее всего меняет работу пользователей и одновременно сложнее всего реализовать технологически.

| № | Что даёт двойник (основное) | Из каких потребностей вытекает | Главная пользовательская ценность |
|---|----------------------------|----------------------------------|-----------------------------------|
| O1 | Единая связанная модель ресурсов, мощностей, кадров, инфраструктуры и рисков | 2.1 Информационные, 2.4 Инфраструктурные, 2.5 Ресурсные | Решение принимается на цельной картине, а не на ведомственных фрагментах |
| O2 | Сценарное моделирование с учётом нескольких слоёв одновременно | 2.2 Управленческие, 2.4 Инфраструктурные, 2.5 Ресурсные | Проверка решения до реализации с проверяемыми допущениями |
| O3 | Связка рынка труда, образования и реальных потребностей отраслей | 2.3 Кадровые и социальные | Снижение одновременных дефицитов кадров и невостребованных специалистов |
| O4 | Пять контуров доступа как встроенное свойство, а не надстройка | 2.6 Безопасность, 2.7 Технологические | Одновременная открытость и защита чувствительных сведений |
| O5 | Суверенный стек данных и моделирования | 2.7 Технологические | Управляемость и переносимость в любой геополитической ситуации |

### 5.2. Где технологически реализация неочевидна

Для каждого «основного» пункта выделены конкретные точки, где сегодня нет готового, проверенного решения. Это и есть зоны риска презентации проекта — их нужно прояснять отдельно.

| № | Основное | Что неочевидно технологически | Почему это сложно |
|---|----------|-------------------------------|-------------------|
| O1 | Единая модель | Согласование классификаторов, паспортов качества и регламентов обновления между десятками систем | Каждая ведомственная система имеет свою модель, цикл обновления и владельца; технических интеграций мало без правовой и регламентной базы |
| O1 | Единая модель | Прослеживаемость «откуда взят показатель» при цепочках агрегаций | Показатель часто получен из нескольких источников разной свежести — сохранить происхождение по всей цепочке расчёта непросто |
| O2 | Сценарии | Корректное моделирование зависимостей между слоями (энергетика, транспорт, связь, кадры, ресурсы) | Графы зависимостей строятся на неполных данных операторов; качество прогноза зависит от закрытых сведений |
| O2 | Сценарии | Версионирование моделей вместе с данными, на которых они обучены | Требует инфраструктуры воспроизводимости и регламента приёмки моделей, который пока не стандартизирован для государственных систем |
| O3 | Кадры/образование | Связка вакансий, компетенций, образовательных программ и реальных потребностей в реальном времени | Источники живут в разных системах с разными классификаторами; «реальная потребность» часто не оцифрована |
| O3 | Кадры/образование | Прогноз дефицита под конкретный муниципалитет и отрасль | Локальная статистика тонкая, шум большой, модели легко переобучаются и теряют точность за пределами обучающих регионов |
| O4 | Контуры доступа | Безопасная передача результатов между контурами без раскрытия сценариев | Требуется доказуемая обезличенность и защита от реидентификации; универсальной методики для всех классов данных нет |
| O4 | Контуры доступа | Подтверждение полномочий пользователя в межведомственной среде | Разные ведомства используют разные системы аутентификации; единый федеративный механизм нужно проектировать отдельно |
| O5 | Суверенный стек | Замещение зрелых зарубежных компонентов геоаналитики и ML без потери качества | Часть отечественных компонентов ещё развивается; критерии «достаточной зрелости» нужно зафиксировать заранее |
| O5 | Суверенный стек | Переносимость компонентов между инфраструктурами | Зависит от соблюдения открытых стандартов на всех уровнях; на практике стандарты часто соблюдаются частично |

### 5.3. Какие шаги могут быть предприняты

Шаги сгруппированы по горизонтам: ближайшие 3 месяца, 3–9 месяцев (горизонт MVP) и далее. Логика — двигаться от регламентов и пилотов к промышленной модели, не подменяя ведомственные системы.

#### Горизонт 0–3 месяца: подготовка и регламенты

1. Зафиксировать целевые классификаторы и требования к паспортам качества данных (общая база для всех ведомств — единая методическая рамка).
2. Согласовать с пилотными регионами список приоритетных слоёв и владельцев каждого слоя.
3. Запустить каталог метаданных с паспортами по каждому подключаемому слою — как «единый источник правды» по происхождению и качеству.
4. Сформировать матрицу контуров доступа: какие данные, в какой детализации и кому доступны (на основе [Тезис 3](../Тезисы.md#тезис-3-пять-контуров-доступа-как-обязательная-архитектура)).
5. Подготовить регламент передачи между контурами: какие выводы и в какой обезличенной форме передаются «наверх» и «вниз» (на основе [правил из примеров обработки](../Примеры%20обработки%20информации%20для%20контуров.md#правила-корректной-передачи-данных-между-контурами)).
6. Согласовать модель ролей и федеративной аутентификации с ведомствами пилотов.

#### Горизонт 3–9 месяцев: MVP на 2–3 пилотных регионах

7. Развернуть базовый стек MVP по [Архитектуре MVP](../MVP.md#архитектура-mvp): PostGIS, объектное хранилище, ETL/ELT, API, веб-карта, журнал аудита.
8. Подключить минимум 10 ключевых слоёв на регион и обеспечить 100% покрытие паспортами качества.
9. Реализовать минимум 3 сценарных модели — логистический риск, кадровый дефицит, инфраструктурный/ЧС-риск (соответствует [критериям готовности MVP](../MVP.md#критерии-готовности)).
10. Запустить открытый и ведомственный контуры в рабочем режиме; инфраструктурный и закрытый — методически описать; исследовательский — на синтетике.
11. Включить версионирование моделей и наборов данных — каждое обновление модели сопровождается фиксацией данных, на которых она обучена.
12. Провести нагрузочные и проверочные тесты по разграничению доступа: «0 критических нарушений разграничения прав» из [метрик MVP](../MVP.md#метрики-эффективности).

#### Горизонт 9–18 месяцев: масштабирование и зрелость

13. Расширить географию и наполнить инфраструктурный и закрытый контуры данными по согласованным регламентам.
14. Запустить федеративную аутентификацию между ведомствами.
15. Сертифицировать первые модели прогнозов (карточки моделей, регламент приёмки).
16. Подключить отраслевые цифровые двойники как поставщиков данных (промышленность, транспорт, энергетика).
17. Зафиксировать стандарт обмена и спецификации API для разрешённых наборов как открытые документы.
18. Опубликовать методику подключения нового региона и нового слоя данных как воспроизводимый процесс.

#### Сквозные шаги (на всех горизонтах)

- Каждое решение оформляется как сценарий с допущениями, ограничениями и журналом расчёта (Тезис 5).
- Безопасность встраивается с первого дня: контуры, ролевой доступ, аудит, обезличивание (Тезис 6).
- Технологический стек поддерживает переносимость и не зависит от единственного поставщика (Тезис 10).
- Любое расширение функциональности проходит через пилот в одном-двух регионах до федерального масштабирования (Тезис 7).

### 5.4. Сводная таблица «потребность → продукт → шаг»

| Потребность (по Тезисам) | Что даёт двойник | Что неочевидно | Ближайший шаг |
|--------------------------|------------------|----------------|---------------|
| Единая картина ресурсов территории (2.1) | Связанная модель ресурсов, мощностей, кадров, инфраструктуры и рисков (O1) | Согласование классификаторов и паспортов качества | Каталог метаданных и единая методическая рамка (шаг 1, 3) |
| Прослеживаемость данных (2.1) | Паспорт качества для каждого слоя | Прослеживаемость по цепочкам агрегации | Версионирование наборов и моделей (шаг 11) |
| Сценарное планирование (2.2) | Быстрая проверка сценариев (O2) | Зависимости слоёв, версионирование моделей | 3 рабочие сценарные модели в MVP (шаг 9) |
| Связь рынка труда и образования (2.3) | Карта дефицитов и подбор треков (O3) | Связка вакансий, компетенций и потребностей в реальном времени | Слой кадров и образования в пилотных регионах (шаг 8) |
| Видимость зависимостей сетей (2.4) | Графы зависимостей (часть O1, O2) | Полнота данных операторов; раскрытие точек отказа | Инфраструктурный контур, методически описанный в MVP (шаг 10) |
| Учёт природных ресурсов (2.5) | Связанные слои природных ресурсов и допустимых нагрузок | Согласование с ФГИС ЕФГИ и Роснедра | Подключение приоритетных ресурсных слоёв (шаг 8) |

| Суверенный стек (2.7) | Отечественный стек, открытые стандарты (O5) | Зрелость отечественных компонентов и переносимость | Стек MVP на отечественных и открытых компонентах (шаг 7) |

## 6. Резюме презентации

| Раздел | Главное сообщение |
|--------|-------------------|
| 1. Принципиальная схема | Это контур принятия решений с однонаправленным обогащением: чем чувствительнее данные, тем меньше деталей выходит наружу |
| 2. Функциональная схема | Десять функциональных блоков связаны через единый каталог и сервис сценариев, безопасность и качество — сквозные процессы |
| 3. Структура | Слои данных связаны между собой; уровней управления пять, технологический стек суверенный и переносимый |
| 4. Сводная таблица | Один и тот же класс данных живёт в нескольких контурах в разной детализации; персональные данные и точные сведения о критической инфраструктуре не обрабатываются нигде |
| 5. Что даёт двойник | Пять основных продуктов, перечень технологически неочевидных мест и три горизонта шагов от регламентов к промышленной зрелости |
