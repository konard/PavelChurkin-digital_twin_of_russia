# Архитектура

Реализация v0.1 — узкий демонстратор открытого контура:

```text
Дескрипторы источников CSV/файл/API открытого контура
  -> ETL-каркас и паспорта датасетов
  -> Сидированный каталог и записи слоёв/объектов
  -> API FastAPI только для чтения и детерминированный движок сценариев
  -> Дашборд React + MapLibre и просмотр отчётов
  -> Журнал аудита с проверкой хеш-цепочки
```

## Сервисы бэкенда

| Идентификатор спецификации | Реализация                                                                         |
| -------------------------- | ---------------------------------------------------------------------------------- |
| S1 Авторизация/RBAC        | Каркас ролей на основе заголовков: гость, гражданин, бизнес, разработчик, оператор |
| S2 Каталог                 | Эндпоинты и фильтры паспортов датасетов                                            |
| S3 Движок сценариев        | Детерминированные демо-запуски на открытых данных с фиксированными версиями        |
| S4 Геосервис               | Эндпоинты слоёв/объектов и тайловый конверт GeoJSON                               |
| S5 Экспорт                 | Экспорт отчётов в Markdown и минимальный PDF                                       |
| S6 Аудит                   | Хеш-цепочка только для добавления в памяти                                         |
| S7 Шлюз                    | Проверки открытого контура и k-анонимности для публичных датасетов                 |

API намеренно сидируется из `data/demo`, чтобы тесты были воспроизводимы и
не зависели от живых эндпоинтов российских открытых данных.

## Границы данных

Возвращаются только записи с `contour=open`. Демо исключает персональные данные,
схемы критической инфраструктуры и сценарии закрытого контура. Записи датасетов
с агрегированным или анонимизированным статусом ПДн должны удовлетворять условию `k_anonymity >= 5`.

## Реальное время: единый WebSocket (целевая архитектура)

Планируемая архитектура обновлений карты в реальном времени. Ключевой принцип
(по итогам ревью): **не создавать отдельное WebSocket-соединение на каждый слой
карты**. Множество соединений на одного пользователя быстро упирается в лимиты
браузера и сервера, усложняет аутентификацию и переподключение. Вместо этого —
**одно мультиплексируемое соединение** на клиента с логической маршрутизацией
сообщений.

```text
Источники (министерства/ведомства)        Ingress            Шина / брокер            Egress
  REST / вебхуки / SFTP / S3 / Kafka  ->  сервис приёма  ->  Kafka / RabbitMQ    ->  WS-шлюз  ->  один WSS на клиента
                                          (валидация,        (топики, разбиение       (Pub/Sub,     (тысячи клиентов)
                                           нормализация)      по географии/слою)       фильтрация)
```

**Единое мультиплексируемое соединение (egress к пользователям).**

- Один WSS-канал на клиента. Клиент **подписывается/отписывается** на топики
  (слои, регионы) сообщениями в рамках этого же соединения — модель Pub/Sub, а не
  соединение-на-слой.
- Идентификатор топика кодирует слой и географию, например
  `layer:vacancies;bbox-tile:z8/x/y`. Сервер шлёт клиенту только те обновления,
  на топики которых он подписан.
- **Географическая фильтрация**: подписка ограничивается видимой областью
  (bounding box) и уровнем зума. Пространство бьётся на тайлы по зуму (по аналогии
  с XYZ-тайлами карты), клиент подписывается на тайлы в видимой области и
  переподписывается при панораме/зуме. Это ограничивает трафик тем, что реально
  видно на экране.

**Ingress (приём данных от источников).** Обновления от министерств и ведомств
принимаются подходящим под источник способом — REST-загрузка, вебхуки, SFTP,
S3-совместимое объектное хранилище или напрямую Kafka. Сервис приёма валидирует,
нормализует и публикует события в шину.

**Брокер сообщений.** Между ingress и egress стоит брокер (Kafka или RabbitMQ):
он разводит потоки по топикам, сглаживает всплески нагрузки, разъединяет
производителей и потребителей и позволяет масштабировать WS-шлюз горизонтально
(несколько инстансов читают из брокера и рассылают своим клиентам).

**Безопасность.**

- **Шифрование**: только WSS поверх TLS (никакого незащищённого `ws://`).
- **Аутентификация на этапе рукопожатия**: JWT или сессионная cookie проверяются
  при установлении соединения; неаутентифицированные подключения отклоняются до
  апгрейда до WebSocket.
- **Проверка Origin/CORS** при рукопожатии, чтобы соединения инициировались
  только с доверенных источников.
- **Валидация входящих сообщений**: строгая схема сообщений подписки/отписки;
  клиент не может подписаться на топики вне своей роли и контура (авторизация на
  уровне топика).
- **Защита от DoS**: ограничение частоты сообщений и числа подписок на клиента,
  таймауты, ограничение размера сообщения, heartbeat/ping-pong для отключения
  «мёртвых» соединений.

Эта схема относится к целевой архитектуре; в демонстраторе v0.1 обновления карты
берутся из сидированных данных и постраничных запросов к открытому API без
реального времени.
