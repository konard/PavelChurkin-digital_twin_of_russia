# S3-совместимое хранилище MinIO (issue #25, п. 3)

Прототип использует **MinIO** как локальное S3-совместимое объектное хранилище.
Это ответ на вопрос из issue #25: «прояснить, как работать с S3 (MinIO), и можно
ли сохранять несколько баз для последующего сравнения по различным параметрам».

## Как это устроено

MinIO поднимается сервисом `minio` в `infra/docker-compose.yml` и полностью
совместим с протоколом Amazon S3 — значит, подходит любой S3-клиент
(`aws s3`, `boto3`, `mc`, SDK на любом языке).

Подключение задаётся переменными окружения в `.env` (см. `.env.example`):

| Переменная            | Назначение                                    |
| --------------------- | --------------------------------------------- |
| `S3_ENDPOINT`         | адрес MinIO (`minio:9000` в compose)          |
| `S3_ACCESS_KEY`       | ключ доступа                                  |
| `S3_SECRET_KEY`       | секретный ключ                                |
| `S3_BUCKET`           | бакет по умолчанию                            |
| `S3_REGION`           | регион (для S3-клиентов, по умолчанию `us-east-1`) |

При запуске без Docker замените имя сервиса `minio` на `localhost`.

### Веб-консоль

MinIO поднимает веб-консоль (по умолчанию порт `9001`): в ней видно бакеты,
объекты и версии. Логин/пароль — те же `S3_ACCESS_KEY`/`S3_SECRET_KEY`.

### Пример: загрузка и чтение объекта через `mc`

```bash
# алиас на локальный MinIO
mc alias set local http://localhost:9000 "$S3_ACCESS_KEY" "$S3_SECRET_KEY"

# загрузить снапшот выгрузки вакансий
mc cp data/demo/vacancies-sample.json local/$S3_BUCKET/snapshots/2026-07-12.json

# посмотреть список снапшотов
mc ls local/$S3_BUCKET/snapshots/
```

Через Python (`boto3`):

```python
import boto3, os

s3 = boto3.client(
    "s3",
    endpoint_url=f"http://{os.environ['S3_ENDPOINT']}",
    aws_access_key_id=os.environ["S3_ACCESS_KEY"],
    aws_secret_access_key=os.environ["S3_SECRET_KEY"],
)
s3.upload_file("report.md", os.environ["S3_BUCKET"], "reports/vacancy-analysis.md")
```

## Можно ли хранить несколько баз для сравнения?

**Да.** Есть два независимых способа, они дополняют друг друга:

1. **Несколько бакетов/префиксов в MinIO.** Каждую выгрузку (снапшот) кладём под
   собственным ключом — например по дате (`snapshots/2026-07-12/…`) или по
   параметру среза (`by-region/…`, `by-domain/…`). Так можно держать сколько
   угодно версий данных и сравнивать их по любым параметрам: дате выгрузки,
   региону, домену, источнику. Включённое версионирование бакета (`mc version
   enable local/<bucket>`) хранит историю изменений одного и того же ключа.

2. **Несколько логических баз в PostgreSQL/PostGIS.** Реляционные данные (паспорта
   датасетов, объекты двойника, запуски сценариев) версионируются полями
   `source_version`/`dataset_version`/`model_version`, поэтому разные версии
   одной сущности сосуществуют и сравниваются запросами. При необходимости можно
   поднять отдельные схемы/базы под каждый снимок.

Для отчёта «Анализ вакансий Работа России» (см. модуль «Отчёты») сравнение
удобно делать так: сохранять сгенерированный Markdown-отчёт и исходную выгрузку
в MinIO по датам, а затем сопоставлять топы (востребованность, зарплаты, города,
даты создания/изменения) между снимками.
